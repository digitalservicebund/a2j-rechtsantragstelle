# ADR: Redis Session Encryption

## Status

- 2026-03-27: Drafted
- 2026-03-XX: Accepted

## Context

Our application stores user data in Redis. By default, Redis does not encrypt data in use and [explicitly recommends handling this via application-side encryption](https://redis.io/docs/latest/operate/rs/security/encryption/#encrypt-data-in-use). To protect user data against against internal and infra-level access, we need to ensure that ut remains unreadable without the specific user's active browser session.

**Key constraints and considerations**

- The application must support environments without JavaScript (noJS), ruling out Web Crypto API for client-side key generation
- We need a smooth migration path for existing active sessions (no immediate data invalidation)
- We need the ability to disable encryption in local development or act as an emergency kill-switch in production

## Decision

We will implement an application-level encryption layer between React Router's session management and Redis.

- **Encryption Standard**: We will use `AES-256-GCM` via Node's native crypto module, as it is the [de-facto industry standard](https://www.bithost.in/blog/tech-3/aes-256-vs-aes-256-gcm-1) for encrypting data with authentication.
- **Key Generation & Scope**: Keys will be generated by the Node server on the user's first request. We will use a unique key per user session (not per individual flow).
- **Key Storage**: The key will be sent to the browser via a dedicated, securely signed, HTTP-only cookie (e.g., `__vaultKey`). It will never be written to Redis or the file system.
- **Feature Toggle**: Encryption will be controlled via an environment variable `ENABLE_SESSION_ENCRYPTION`.
- **Migration Strategy**: The decryption method will transparently handle both encrypted and unencrypted data. Previously un-encrypted legacy sessions will be encrypted on their next write.

## Consequences

### Positive

- **High Security**: A compromised Redis instance yields only useless binary data. The decryption key exists only in the user's browser cookie (and temporarily in the server's RAM).
- **Seamless UX / NoJS Support**: Because the server handles generation and the browser automatically handles the HTTP-only cookie, the frontend codebase requires zero changes and works perfectly without JavaScript.
- **Zero-Downtime Migration**: Old plaintext sessions remain valid and are seamlessly encrypted upon their next write.

### Negative / Risks:

- **Data Loss on Cookie Clearing**: If a user clears their cookies or the cookie expires, the orphaned data in Redis becomes permanently unrecoverable (though it will be naturally deleted by the Redis TTL).
- **CPU Overhead**: Symmetrical encryption/decryption on every Redis read/write will introduce a marginal increase in CPU overhead on the Node servers.

### Future Considerations:

- **Key Rotation**: Sequential key rotation to support scenarios where a session key needs to be cycled without losing access to historical data could be supported. This will be expanded as needed.
